# App Privacy(プライバシー栄養ラベル)申告内容の整理

App Store Connect の「App のプライバシー」質問票に回答するための整理メモ。
提出時にこのドキュメントの表を転記すれば申告が完了する状態を目指す。

作成日: 2026-07-09

> ⚠️ AdMob(Google Mobile Ads SDK)の収集内容は SDK バージョンによって変わる。
> 提出直前に Google 公式のデータ開示ガイド
> (https://developers.google.com/admob/ios/data-disclosure)で最新の推奨申告を再確認すること。
> GMA SDK 11.x 以降はプライバシーマニフェスト(`PrivacyInfo.xcprivacy`)を同梱しているので、
> Xcode の「App Privacy Report(アーカイブ→ Generate Privacy Report)」でも自動集計を確認できる。

---

## 1. 前提: このアプリで「収集」にあたるものは何か

Apple の定義では、**「デバイス外に送信され、開発者または第三者パートナーがアクセスできる
データ」だけが『収集』**にあたる。このアプリは開発者のサーバーを持たない BYO-AWS 構成
なので、アプリ本体の機能はほぼ申告対象にならない:

| データ | 送信先 | 申告 | 理由 |
|---|---|---|---|
| 写真(+撮影日時等のメタデータ) | 利用者自身の S3 | **対象外** | 開発者はバケットにアクセスできない(利用者管理のAWS) |
| メールアドレス・パスワード | 利用者自身の Cognito | **対象外** | 同上。認証基盤ごと利用者のAWS内 |
| アップロード履歴・重複スキップ記録・接続先設定 | 端末内のみ(UserDefaults/Keychain) | **対象外** | デバイス外に出ない |
| QRスキャン時のカメラ映像 | どこにも送信されない | **対象外** | 読み取り結果(設定JSON)を端末に保存するだけ |
| **AdMob が収集するデータ** | Google | **申告対象** | 第三者パートナーによる収集。下の表がこのアプリの申告のほぼすべて |

**審査員向けの説明文例(Review Notes に貼る用)**:
「本アプリは開発者運営のサーバーを持ちません。写真および認証情報は、利用者自身が
自分の AWS アカウント内に構築したバックエンド(S3 / Cognito)へ直接送信され、
開発者はアクセスできません。したがって App Privacy の申告は広告 SDK(Google AdMob)
の収集分のみです。」

## 2. 先に決めること: トラッキング(ATT)の方針

申告の分岐点は「パーソナライズ広告を出すかどうか」の一点。

### 案A: 非パーソナライズ広告(NPA)のみ — **初回リリースはこちらを推奨**

- 「トラッキング」= **なし** で申告できる。ATT(許可ダイアログ)の実装が不要
- 広告収益の単価は下がるが、申告・実装・審査がすべてシンプル
- **アプリ側は実装済み(2026-07-09)**: 全リクエストが `AdsConfig.makeRequest()` 経由で
  `npa=1`(`GADExtras`)を付与して送信される。新しい広告枠を足すときも必ずこれを使うこと
- **残り(AdMob本番化のときに)**: AdMob 管理画面の「EU ユーザーの同意」設定を
  NPA 運用に合わせる(または UMP SDK 導入を検討)

### 案B: パーソナライズ広告(IDFA 利用)

- 収益単価は上がるが、以下がすべて必要になる:
  - ATT ダイアログ実装 + `NSUserTrackingUsageDescription`
  - 「トラッキング」= **あり**(デバイスID)で申告
  - UMP SDK による同意取得(GDPR 圏に配信する場合)
- 拒否されたら結局 NPA 配信になるため、小規模アプリでは費用対効果が薄い

以下の表は **案A(NPA のみ・トラッキングなし)** を前提に書く。
案Bにする場合は「トラッキングに使用」列の識別子を「はい」に変え、ATT を実装すること。

## 3. App Store Connect への申告内容(案A前提)

質問1「このアプリまたはサードパーティパートナーは、このアプリからデータを収集しますか?」
→ **はい、データを収集します**(AdMob があるため「いいえ」にはできない)

収集するデータタイプ(すべて AdMob / Google Mobile Ads SDK 由来):

| Apple のカテゴリ | データタイプ | ユーザーに紐付く | トラッキングに使用 | 利用目的(選択肢) |
|---|---|---|---|---|
| 識別子 | デバイス ID | はい | いいえ | 第三者の広告、アプリの機能(不正防止・頻度制御) |
| 使用状況データ | 広告データ | はい | いいえ | 第三者の広告 |
| 使用状況データ | 製品の操作 | はい | いいえ | 第三者の広告、分析 |
| 位置情報 | おおよその位置情報(IP由来) | はい | いいえ | 第三者の広告、不正防止 |
| 診断 | クラッシュデータ | いいえ | いいえ | アプリの機能(SDKの安定性) |
| 診断 | パフォーマンスデータ | いいえ | いいえ | アプリの機能 |
| 診断 | その他の診断データ | いいえ | いいえ | アプリの機能 |

質問「トラッキング」→ **いいえ**(案Aの場合。NPA設定の実装が前提!)

- 「ユーザーに紐付く」= デバイスIDをキーに Google 側で紐付くため、識別子・使用状況・位置は「はい」で申告するのが Google の推奨に沿う(診断系は紐付けなし扱いが一般的)
- 「収集したデータ」はアプリ本体の機能(メール・写真)を **含めない**(§1 の理由)

## 4. あわせて必要な整合チェック

- [x] プライバシーポリシーに AdMob の収集について記載済み(`notes/privacy-policy.md` §第三者サービス)
- [x] ポリシーの「アカウント削除」をアプリ内削除の実装に合わせて更新(2026-07-09)
- [x] 案Aで行く場合: **NPA 設定の実装** — アプリ側は実装済み(§2)。AdMob 管理画面側の設定のみ本番化時に
- [ ] App Store Connect の「プライバシーポリシー URL」に GitHub Pages の
      `https://<ユーザー名>.github.io/photouploader/privacy.html` を設定(Pages 有効化が先)
- [ ] 提出直前に Google のデータ開示ガイドと Xcode の Privacy Report で最終確認

## 5. App Store Connect での入力手順

1. App Store Connect → 対象アプリ → サイドバー「App のプライバシー」
2. プライバシーポリシー URL を入力(§4)
3. 「開始」→ データ収集「はい」
4. §3 の表のデータタイプにチェック → タイプごとに「利用目的」「ユーザーへの紐付け」「トラッキング」を表のとおり回答
5. 公開前にプレビューで「ユーザーのトラッキングに使用されるデータ: なし」になっていることを確認(案A)
